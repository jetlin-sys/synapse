# 多智能体研发会议室实现方案

本文档说明如何在 **Synapse 现有能力**上落地「智能小鲸主导、多研发智能体分工、系统固定 SKILL、小鲸验收与纠偏」的**研发会议室**形态，并给出分阶段实施路径与代码落点。

> 架构背景与多 Agent 总览见：[多 Agent 架构](../multi-agent-architecture.md)。

---

## 1. 目标与边界

### 1.1 业务目标

| 角色 | 职责 |
|------|------|
| **智能小鲸（主导）** | 理解用户意图与当前场景；拆解任务；通过委派驱动其他智能体；对子体产出做**结果与格式**验收；不通过则调整任务说明或改派；面向用户汇总与对齐预期。 |
| **研发类智能体（Worker）** | 在**系统给定**的技能与工具边界内执行具体子任务；输出结构化、可验收的答复。 |
| **系统** | 为每个 Worker 绑定**固定 SKILL 集合**（及可选工具/MCP 白名单），避免子体随意扩权或乱选技能。 |

### 1.2 设计边界（推荐）

- **不另起**与 `AgentOrchestrator` / `AgentFactory` 并行的「第二套会议室运行时」，除非后续需要完全脱离 LLM 的流程图引擎（本方案不覆盖该路线）。
- **优先**用 **协调者模式 + AgentProfile 画像 + 提示词契约** 满足 80% 场景；结构化校验与专用入口在第二阶段以后按需加代码。

---

## 2. 与当前 Synapse 的映射

会议室中的角色与 Synapse 模块对应关系如下。

| 会议室概念 | Synapse 机制 | 主要代码 / 配置 |
|------------|--------------|-----------------|
| 小鲸主导、规划与委派 | `role=coordinator` + `coordinator_mode_enabled` | `src/synapse/agents/profile.py`（`role`）；`src/synapse/config.py`（`coordinator_mode_enabled`）；`src/synapse/core/reasoning_engine.py`（coordinator 分支）；`src/synapse/prompt/builder.py` + `src/synapse/agents/coordinator_prompt.py` |
| 研发体分工、并行/串行 | `delegate_to_agent` / `delegate_parallel` 等 | `src/synapse/tools/handlers/agent.py`；`src/synapse/agents/orchestrator.py` |
| 系统携带固定 SKILL | `skills_mode=inclusive` + `skills=[...]` | `src/synapse/agents/profile.py`（`SkillsMode`） |
| 系统约束工具 / MCP（可选） | `tools`/`tools_mode`、`mcp_servers`/`mcp_mode` | 同上 `AgentProfile` |
| 子体不能再委派 | 子 Agent 系统提示中禁用委派类工具 | `src/synapse/core/agent.py`（`_build_multi_agent_prompt_section`） |
| 会话级实例与池化 | 按 session + profile 复用子体实例 | `src/synapse/agents/factory.py`（`AgentFactory`、`AgentInstancePool`） |
| 全局多 Agent 总开关 | `multi_agent_enabled`（默认已开） | `src/synapse/config.py` |

**结论**：会议室 = **协调者画像（小鲸）** + **多个 INCLUSIVE Worker 画像（研发体）** + **协调者模式配置** + **（可选）输出契约与后置校验**。

## 3. 分阶段落地路径

### 阶段 A — 配置与提示词（首选 MVP）

**目标**：在不改或少改代码的前提下跑通「会议室」协作闭环。

1. **打开协调者模式**  
   - 将 `coordinator_mode_enabled` 设为 `true`（与全局/组织配置方式一致，参见 [配置说明](../configuration.md)）。  
   - 确保 `multi_agent_enabled` 为 `true`（当前仓库默认即为 `true`）。

2. **创建「智能小鲸」画像**  
   - `role`: `"coordinator"`。  
   - `custom_prompt`: 写明会议室职责——包括但不限于：  
     - 委派前把用户目标与约束写进**自包含**的 Worker 指令（协调者提示中已强调 Worker 看不到主会话）。  
     - 收到 Worker 返回后：按检查清单验收**完整性、格式、是否答非所问**；再决定汇总用户或打回重派。  
   - 按需收窄小鲸的工具集（协调者模式下本身以委派/规划为主，具体以权限与工具注册为准）。

3. **为每个研发方向创建 Worker 画像**  
   - `role`: `"worker"`（或默认 worker 语义）。  
   - `skills_mode`: `"inclusive"`。  
   - `skills`: 仅列出该角色允许使用的技能 id（与 `SKILL.md` / 技能注册名一致）。  
   - 可选：`tools_mode`/`mcp_mode` 设为 `inclusive` 并列出允许项，实现「系统控权」。

4. **强化协调者规则（二选一或组合）**  
   - 在 `src/synapse/agents/coordinator_prompt.py` 中增加「研发会议室」专节（影响所有 coordinator）；或  
   - 仅在小鲸画像的 `custom_prompt` 中写会议室专规（**推荐**：不影响其他 coordinator 画像）。

5. **会话入口**  
   - 默认会话主 Agent 选用小鲸画像；小鲸通过委派工具选择对应 `profile_id` 的研发体。

**阶段 A 验收标准**：小鲸能并行/串行委派；各 Worker 仅能加载其 `skills` 白名单内的技能；子体无法再次委派；用户侧只看到小鲸的汇总与进展说明。

---

### 阶段 B — 输出契约与（可选）程序化校验

**目标**：降低「仅靠模型自觉验收」的方差。

1. **委派消息内嵌固定输出模板**  
   - 要求 Worker 使用统一结构返回（示例）：背景摘要、结论、依据/引用、风险与假设、需小鲸向用户确认的事项。  
   - 可要求关键字段使用 **JSON** 或固定 **Markdown 小节标题**，便于小鲸与人类阅读。

2. **（可选）在编排层增加轻量校验**  
   - 在 `src/synapse/agents/orchestrator.py` 中，于将 `DelegationResult` 交回主 Agent 之前，增加可选步骤：  
     - JSON Schema / 必填标题检测；  
     - 失败时将「校验错误」作为内部上下文附加给协调者下一轮，而不是直接暴露给用户。  
   - 该步骤应**可配置**（按 profile、按会话或按组织开关），避免影响通用多 Agent 行为。

**阶段 B 验收标准**：明显格式缺失时系统能自动提示协调者重派，而非把残缺结果当成功交付。

---

### 阶段 C — 产品化「会议室」入口

**目标**：固定阵容、可视化与运营配置。

1. **预设阵容**  
   - 在 `src/synapse/agents/presets.py` 或组织级配置中增加「会议室」预设：小鲸 + 若干 `profile_id`。  
   - 或在前端（如 `apps/setup-center`）增加「一键进入会议室」并绑定默认画像组合（具体 UI 另立需求）。

2. **可观测性**  
   - 利用编排器已有子 Agent 状态（如 `SubAgentStatus`）做列表展示与超时提示（见 `src/synapse/agents/orchestrator.py`）。

**阶段 C 验收标准**：非技术人员可通过固定入口发起会议室会话，且能感知各子任务状态。

## 4. 配置项清单（实施时核对）

| 项 | 说明 |
|----|------|
| `multi_agent_enabled` | 多 Agent 总能力，应为开启状态。 |
| `coordinator_mode_enabled` | 为 `true` 时，`role=coordinator` 的画像走协调者工具集与规划约束。 |
| 小鲸画像 `role` | 必须为 `coordinator`。 |
| Worker 画像 `skills_mode` | 建议使用 `inclusive`。 |
| Worker 画像 `skills` | 与仓库/全局技能 id 严格一致，避免拼写错误导致技能未加载。 |
| 小鲸 `custom_prompt` | 会议室流程、验收清单、与用户沟通风格。 |

画像持久化位置与 API 细节见 [多 Agent 架构](../multi-agent-architecture.md) 中 ProfileStore 相关描述。

## 5. 提示词与协作规范（建议写入小鲸 `custom_prompt`）

以下可作为 `custom_prompt` 附录的提纲（实施时按产品语言微调）。

1. **委派前**：复述用户目标中的硬约束（环境、版本、不可改动范围）；每个 Worker 一条自包含指令。  
2. **并行策略**：只读/调研类可并行；同一文件或同一数据源的写入互斥（与 `coordinator_prompt.py` 中原则一致）。  
3. **验收**：检查约定字段是否齐全、格式是否符合模板、结论是否直接回应用户问题；不通过则**明确**告知 Worker 缺什么、如何补。  
4. **对用户**：不暴露内部扯皮细节；进展与最终结论分层说明。

## 6. 风险与缓解

| 风险 | 缓解 |
|------|------|
| Coordinator 提示过长挤占 token | 会议室专规放在小鲸 `custom_prompt`，公共 `coordinator_prompt.py` 保持精简。 |
| Worker 仍输出非结构化内容 | 阶段 B 模板 + 可选 orchestrator 校验。 |
| 技能 id 与仓库不一致 | 在 CI 或发布脚本中对「会议室画像」做技能存在性检查（自定义脚本即可）。 |
